# Filesystem semantics

Protocol version 5 supports a read-only-by-default or explicitly authorized read/write export. Ordinary files and directories implement Finder/application operations including arbitrary positioned I/O, create/delete, truncate, metadata, namespace changes, symlinks, hardlinks, locks, allocation, statfs, sync, and metadata-bearing readdirplus. File revisions are change indicators, not globally ordered versions; a multi-request read rejects mixed revisions.

Names are lossless Unix bytes in the protocol, client core, and Unix server. macOS lookup first tries exact bytes, then applies Unicode NFD/case-insensitive fallback only when both names are valid UTF-8; ambiguous folded names fail instead of selecting the wrong file. Modern macOS filesystems and macFUSE reject an invalid UTF-8 pathname before it reaches the adapter, so a macOS mount cannot expose such a backing name even though Linux and the wire format preserve it. Stable local macFUSE inode numbers map to persisted opaque server nodes. `forget` and `batch_forget` evict zero-reference translations and release the authenticated server session's registry capacity.

Extended attributes support size probes, ranged reads, create/replace/upsert writes, listing, and removal. Finder tags, quarantine metadata, and custom app xattrs use these operations. A persistent Linux server maps logical names into its `user.*` namespace; `com.apple.ResourceFork` uses an owner-private sidecar with a 64 MiB limit. Backup time and volume name are reported through the macOS-specific callbacks.

Server-side copy tries Linux reflink for an eligible whole-file clone, then `copy_file_range`, then a bounded buffered fallback. Same-file overlaps have memmove semantics. `SEEK_DATA`/`SEEK_HOLE` is available on macOS through its vnode seek ioctls; safe `FIONREAD`, regular-file poll readiness, and virtual logical `bmap` are also wired through macFUSE. Arbitrary ABI-dependent ioctls return `ENOTTY`.

The adapter implements and capability-gates `copy_file_range`, metadata-bearing `readdirplus`, and `exchangedata`. The macFUSE 5.3 kernel backend tested on macOS 15.1 advertises none of those optional messages, so native `clonefile(2)` falls back or returns `ENOTSUP`, ordinary `readdir` uses QuickFS's metadata-bearing remote directory snapshot without extra remote `getattr` calls, and `exchangedata(2)` returns `ENOTSUP`. [macFUSE explicitly dropped `exchangedata` on macOS 11](https://macfuse.github.io/2020/10/30/macfuse-4.0.0.html). The QuickFS protocol/client/server operations remain directly usable and transport-tested, and the callbacks remain available for a bridge that advertises them. [FSKit requires macOS 15.4 or later](https://github.com/macfuse/macfuse/wiki/FUSE-Backends) and was therefore unavailable on the tested host.

FIFO/socket and privileged device entries are available when the backing daemon is built for Linux and its OS permissions allow `mknod`; a development daemon running on macOS advertises that feature as unavailable. Absolute symlink targets and targets containing `..` are intentionally rejected to preserve export confinement. Xattrs on the symlink object itself are also rejected because portable descriptor-relative Unix APIs cannot address them without a path race.

Authenticated reconnect accepts only the same persisted server epoch. Revision-matched handles are reopened and live-client lock state is replayed after a daemon restart. Locks are not immortal disk state: closing/disconnecting the owning client releases them.

The persistent cache is read-through and read-only while disconnected. It can serve previously authenticated metadata, directory snapshots, symlink targets, statfs results, and revision-keyed byte ranges. It does not cold-start a mount while the server is unavailable and never queues offline mutations. See [caching](caching.md) for why these are deliberate consistency and authorization boundaries rather than missing cache toggles.
