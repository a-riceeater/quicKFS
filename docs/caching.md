# Caching

The mount uses a bounded persistent read-through cache for metadata, directory snapshots, symlink targets, filesystem statistics, and byte ranges keyed by node plus exact file revision. Namespaces bind the authenticated certificate fingerprint, persisted server epoch, and username. Payloads are SHA-256 checked, atomically written, owner-private, protected by a namespace lock, and LRU-evicted under a hard byte budget. SHA-256 uses the platform's optimized assembly implementation where available.

Metadata, directory snapshots, symlink targets, and filesystem statistics have an in-memory front tier. File data additionally uses a bounded 256 MiB process-local hot range tier. A cold persistent block is read and integrity-checked once, inserted into that tier, and then sliced locally for later small FUSE reads instead of rereading and rehashing the entire block.

After an online mount loses connectivity, already cached immutable revisions remain readable. Arbitrary and overlapping requests are assembled from cached ranges. Sequential and copy-sized online reads use the configured read-ahead block (16 MiB by default), clamped to the negotiated server/client limit. Requests smaller than 1 MiB use at most a 1 MiB aligned block, preventing Finder/header/thumbnail probes from turning into a cold 16 MiB RAID read. Missing independent blocks are concurrent, while overlapping readers for the same node revision and aligned block enter single-flight before persistent lookup; they share one disk verification or one remote fetch and observe the same failure instead of serially retrying/reconnecting. Offline read-only handles permit harmless flush/sync/close so an application that successfully consumed cached data does not receive a false close error.

## Adaptive sequential read-ahead

A single sequential read fails to fill a high-latency link because macFUSE issues sequential access as independent `read` callbacks: each stalls a full round trip before the next begins, so only one range request is ever in flight and the pipe sits mostly idle. quicKFS therefore keeps a *window* of speculative block fetches in flight ahead of the consumer. Each open handle owns a per-handle controller that watches its access pattern; once a sequential run is established it fetches blocks past the current read through the same revision-keyed single-flight gate as demand reads, so the next kernel `read` finds the block cached or joins the in-flight fetch instead of paying another round trip. Speculation and demand for the same block coalesce; a mutation bumps the file revision and orphans any speculatively fetched blocks, so read-after-write coherence is preserved.

The window is not a constant. The number of concurrent range requests that saturates a link depends on its bandwidth-delay product, which varies by location, network, and even minute to minute — a sub-millisecond LAN needs about one while a high-RTT WAN needs many. The controller therefore hill-climbs the window against *measured delivered throughput*: it grows while reads are served faster, holds when throughput plateaus (the link is saturated), and backs off on regression or error. Delivered throughput is measured directly on the demand-read path — a too-small window leaves demand reads stalling on the network, and a window that covers the pipe turns them into prefetched cache hits — so no separate bandwidth probe is needed. On a low-latency LAN a single in-flight request already saturates the link, so the window converges to roughly one and adds no speculative load; random or seeky access disables speculation entirely so it never wastes bandwidth or cache. Growth is bounded by a per-mount memory ceiling (`--read-ahead-max-bytes`, 64 MiB by default, `0` to disable), which also keeps a single client under the server's in-flight read budget; a global permit pool sized to that ceiling bounds total speculative bytes in flight and guarantees an interactive demand read never queues behind a wall of speculation. Speculative fetches are scheduled at the same granularity the demand path chose for the triggering read — the `--cache-block-kib` block for copy-sized reads, the bounded 1 MiB block for sub-1 MiB reads — so speculative and demand fetches always coalesce in single-flight, and a stream of small kernel reads (Preview and Quick Look read images this way) or a file smaller than the large block still gets read-ahead instead of one serial round trip per megabyte. This is entirely a client-side scheduling change — the wire protocol and server are unchanged.

## Persistent write-behind: group commit and backpressure

Every durable cache mutation used to rewrite the whole manifest — clone the full entry map, serialize it to JSON, fsync it and the directory — while holding the state mutex that every cache lookup also needs. That is O(total entries) work per stored block; once a mount accumulates tens of thousands of entries (a browsed media tree easily reaches a multi-megabyte manifest), a burst of read-through stores from read-ahead plus a Finder thumbnail sweep floods the single cache-writer thread with hours of backlog, pins a CPU on manifest serialization, and starves every read of the state lock — the filesystem appears hung while the network is idle.

Three mechanisms keep the writer proportional to the work:

- **Group commit.** The writer thread drains up to a batch (256 jobs) from its queue, applies them with manifest writes deferred, then writes the manifest once for the whole batch — obsolete payload files are also removed only after that write, preserving the manifest-first crash ordering. Waited operations (`enqueue_and_wait`) complete only after their batch's flush, so completion still implies durability.
- **Cheap stores.** Payload hashing and the atomic payload-file write happen before the state lock is taken (file names are content-addressed, so this is safe under concurrency), and a store whose manifest entry is already current is recognized from the manifest alone and skips the clone and commit entirely.
- **Backpressure.** Opportunistic read-through stores are dropped — the in-memory tiers still serve them — when the writer backlog exceeds 1,024 jobs or 128 MiB of queued range payloads, so the queue's latency and memory stay bounded. Coherence-critical work (invalidations, waited inserts) is never dropped.

An enriched directory snapshot embeds each child's metadata. Content, attribute, xattr, link, rename/move, exchange, and remove operations therefore invalidate not only the changed node's own metadata/ranges but every cached directory snapshot containing that node. The persistent implementation commits that node-wide invalidation in one manifest transaction. The adapter applies the same rule to active Finder views and retains metadata for the same 30-second lifetime as the directory snapshot, so a second callback wave cannot fall back to one remote `GetMetadata` request per child.

The cache is not an offline synchronization system. Mount startup still authenticates the server and selects a trustworthy epoch/authorization namespace. Write opens and every namespace/content mutation fail closed while disconnected. No operation journal, queued mutation, merge policy, lock lease, or conflict UI exists.

This is intentionally similar to an ordinary SMB client's read/page cache, not to a synchronization product. SMB is not wholly cache-free: clients use caching, opportunistic locks/leases, and durable handles, while optional products such as Windows Offline Files can add disconnected synchronization. A basic share still needs its server for authoritative namespace and mutation decisions.

Allowing QuickFS to mutate while disconnected would create exactly the merge risks that distinction suggests. Two clients could edit overlapping byte ranges, rename or delete the same object, reuse a pathname, or believe they each hold an exclusive lock. The server might also revoke access, change quota, or begin a new export epoch while a client is away. A safe implementation would need a durable idempotent mutation journal, base revisions and preconditions, stable object identity, reauthentication plus permission/quota/lock revalidation, defined rename/delete conflict rules, and user-visible reconciliation. QuickFS deliberately fails writes closed until such a design exists.

Cold-start offline mounting is a separate problem. Without contacting the server, a client cannot authenticate the current server identity and account, confirm the export epoch, learn that permissions were not revoked, or prove that its cached namespace is current. Serving selected data only after a previously authenticated online mount disconnects keeps stale cache entries from becoming an unauthenticated substitute filesystem.
