# Caching

The mount uses a bounded persistent read-through cache for metadata, directory snapshots, symlink targets, filesystem statistics, and byte ranges keyed by node plus exact file revision. Namespaces bind the authenticated certificate fingerprint, persisted server epoch, and username. Payloads are SHA-256 checked, atomically written, owner-private, protected by a namespace lock, and LRU-evicted under a hard byte budget. SHA-256 uses the platform's optimized assembly implementation where available.

Metadata, directory snapshots, symlink targets, and filesystem statistics have an in-memory front tier. File data additionally uses a bounded 256 MiB process-local hot range tier. A cold persistent block is read and integrity-checked once, inserted into that tier, and then sliced locally for later small FUSE reads instead of rereading and rehashing the entire block.

After an online mount loses connectivity, already cached immutable revisions remain readable. Arbitrary and overlapping requests are assembled from cached ranges. Sequential and copy-sized online reads use the configured read-ahead block (16 MiB by default), clamped to the negotiated server/client limit. Requests smaller than 1 MiB use at most a 1 MiB aligned block, preventing Finder/header/thumbnail probes from turning into a cold 16 MiB RAID read. Missing independent blocks are concurrent, while overlapping readers for the same node revision and aligned block enter single-flight before persistent lookup; they share one disk verification or one remote fetch and observe the same failure instead of serially retrying/reconnecting. Offline read-only handles permit harmless flush/sync/close so an application that successfully consumed cached data does not receive a false close error.

## Adaptive sequential read-ahead

A single sequential read fails to fill a high-latency link because macFUSE issues sequential access as independent `read` callbacks: each stalls a full round trip before the next begins, so only one range request is ever in flight and the pipe sits mostly idle. quicKFS therefore keeps a *window* of speculative block fetches scheduled ahead of the consumer. Once a sequential run is established, blocks past the current read are fetched through the same revision-keyed single-flight gate as demand reads, so the next kernel `read` finds the block cached or joins the in-flight fetch instead of paying another round trip. Speculation and demand for the same block coalesce; a mutation bumps the file revision and orphans any speculatively fetched blocks, so read-after-write coherence is preserved.

**Multiple streams per handle.** macOS multiplexes independent readers over one FUSE file handle: a media player's page-ins arrive interleaved with Spotlight, Quick Look, and the player's own index probes, all carrying the same `fh`. Each handle therefore tracks up to four concurrent sequential streams, matching every read to the stream it continues (nearest expected offset, within one block of slack for kernel reordering). A read that continues no tracked stream recycles the least-recently-used slot and starts cold, so genuinely random access never speculates — and one reader's seeks never destroy another reader's window. An earlier single-stream design treated every interleave switch as a seek and zeroed its window; live traces during video playback showed the controller being reset dozens of times per minute and speculation never surviving.

**Stall-driven window sizing.** The window grows exactly when a demand read actually blocked on the network for at least 10 ms — direct evidence the window does not cover the link's bandwidth-delay product — doubling at most once per half-window of delivered bytes so a single long stall cannot balloon it, and it holds while demand reads are served from prefetched data (a resting window costs only bounded memory, and jitter absorption is worth far more). An earlier design hill-climbed on throughput measured over demand-blocked time; once reads became prefetch hits that signal was pure noise and the window whipsawed between one and four blocks forever instead of holding. Growth is bounded by a per-mount memory ceiling (`--read-ahead-max-bytes`, 64 MiB by default, `0` to disable), which also keeps a single client under the server's in-flight read budget.

**Cold-start priming.** The adaptive window only arms after two in-order reads, and used to arm at a single block — which left the opening reads of a freshly opened file with no read-ahead at all, so each cold block was demand-fetched serially at its full per-block latency. On a low-latency LAN that serial ramp, not bandwidth, is what stalled the start of playback: a paced 8 MB/s stream reading a cold region stalled 150 ms–1 s at the very start, rising to ~1 s when a Finder directory crawl shared the one QUIC connection. Two changes remove it. The first read of a handle now primes a shallow speculative read-ahead past the demand block, so the second read is a cache hit rather than another serial cold fetch; and once a sequential run is confirmed the window arms several blocks deep instead of one, so it is already a cushion the moment speculation begins. Priming is gated on there being no tracked stream yet, so only the genuine cold open primes — a mid-file seek that lands beside a still-live stream, and the many short-lived reads of a metadata/thumbnail crawl, start cold exactly as before, which bounds the wasted read-ahead of a one-shot probe (and the range cache reclaims it anyway). Measured against the same paced first-open workload, both the isolated and crawl-contended stalls above drop to zero, with per-read latency staying under ~75 ms — only the unavoidable first-block cold fetch of the bytes actually requested remains.

**Buffer depth is not fetch concurrency.** The window says how far ahead of the consumer data should already be cached; it deliberately does not say how many transfers run at once. Fetching a whole window concurrently runs the link as a convoy — QUIC fair-shares bandwidth, so every block's completion time becomes `in-flight bytes ÷ link rate`, and a demand miss during the convoy waits that long (live-traced at 64 × 1 MiB in flight: every fetch took 1–3.5 s and playback froze for exactly that). Speculation is therefore bounded four ways: a global in-flight **byte** pool (denominated in 64 KiB units, so 1 MiB-granularity streams can use the whole budget rather than being capped at `budget ÷ block_size` fetches); a global cap of 8 concurrent speculative fetches, enough to saturate any realistic link at 1 MiB granularity while keeping each individual transfer fast; and a per-read scheduling burst limit of 8 blocks. The in-flight byte pool is deliberately the *convoy* ceiling, **not** the buffer-depth ceiling: it is capped at 32 MiB, well under `--read-ahead-max-bytes` (which still governs how deep the window may grow, and so how much prefetched data may sit *cached* ahead of the consumer). This decoupling matters for large cache blocks. The 8-fetch cap alone was sized for 1 MiB blocks (≈8 MiB in flight); with `--cache-block-kib 8192` (8 MiB blocks) it would instead put 8 × 8 = 64 MiB of speculation on the wire, and under concurrent load — Spotlight or Quick Look scanning other files while a video plays over the same connection — a playback demand read stalled hundreds of milliseconds behind that convoy (reproduced: paced playback under six bulk readers spiked to ~700 ms; capping the convoy at 32 MiB dropped the worst case to ~130 ms, often zero, with no loss of single-stream throughput). At 1 MiB granularity the 8-fetch cap still binds first, so those streams are unchanged. Both budgets are reserved with a non-blocking `try_acquire` at schedule time — a block whose budget is unavailable is unscheduled (the cursor rewinds) rather than parked, because a parked fetch counts as in-flight without fetching and leaves a never-fetched hole in the window that later surfaces as a multi-second demand stall. Each speculative fetch also carries a hard 120 s deadline so a fetch wedged behind a dying connection can never pin the slot pool, and a demand read that single-flight-joined it takes over the fetch instead of inheriting an unbounded wait. Demand reads never take permits, so speculation can never starve interactive I/O.

Speculative fetches are scheduled at the same granularity the demand path chose for the triggering read — the `--cache-block-kib` block for reads larger than 1 MiB, the bounded 1 MiB block for reads of 1 MiB and below (macFUSE commonly delivers exactly-1 MiB reads, and serving those from 16 MiB blocks made every prefetch miss a whole-large-block wait) — so speculative and demand fetches always coalesce in single-flight, and a stream of small kernel reads (video players and Preview read this way) or a file smaller than the large block still gets read-ahead instead of one serial round trip per megabyte. This is entirely a client-side scheduling change — the wire protocol and server are unchanged. `QUICKFS_PREFETCH_DEBUG=1` streams controller decisions, growth events, and stall/starvation diagnostics to stderr.

## macOS kernel read-ahead is disabled on the volume

The mount passes `-o noreadahead`, delegating all read pipelining to the adaptive prefetcher above. The macOS kernel's own speculative read machinery is tuned for microsecond-latency local disks and is actively harmful on a network filesystem: investigating stuttering video playback (2 GB file, LAN, playback needs ~4 MB/s against a ~50 MB/s link) showed the kernel racing 100+ MB ahead of the paced consumer in 16 KiB page-ins, having those speculative pages evicted before the player reached them, then re-reading the same regions in repeated multi-hundred-megabyte sweeps — 5.5× read amplification — while the player's own page-in queued behind the storm for 2–4 seconds at a time, with the FUSE layer answering every individual request in under a millisecond. The stall offsets doubled (8, 16, 32, 64 MB into any run — the kernel read-ahead window ramping), and the same paced workload with kernel read-ahead disabled per-fd (`F_RDAHEAD = 0`) or with the page cache bypassed (`F_NOCACHE`) had zero stalls. With `noreadahead` volume-wide plus the client prefetcher: zero stalls in 60 s paced playback at 4 and 8 MB/s on cold data, and flat-out cold sequential throughput roughly doubled (26.8 → 49.8 MB/s on the same link) because the client window streams smoothly instead of convoying. The page cache itself (and mmap) remain enabled; only the kernel's speculative read-ahead is off.

## Persistent write-behind: group commit and backpressure

Every durable cache mutation used to rewrite the whole manifest — clone the full entry map, serialize it to JSON, fsync it and the directory — while holding the state mutex that every cache lookup also needs. That is O(total entries) work per stored block; once a mount accumulates tens of thousands of entries (a browsed media tree easily reaches a multi-megabyte manifest), a burst of read-through stores from read-ahead plus a Finder thumbnail sweep floods the single cache-writer thread with hours of backlog, pins a CPU on manifest serialization, and starves every read of the state lock — the filesystem appears hung while the network is idle.

Three mechanisms keep the writer proportional to the work:

- **Group commit.** The writer thread drains up to a batch (256 jobs) from its queue, applies them with manifest writes deferred, then writes the manifest once for the whole batch — obsolete payload files are also removed only after that write, preserving the manifest-first crash ordering. Waited operations (`enqueue_and_wait`) complete only after their batch's flush, so completion still implies durability.
- **Cheap stores.** Payload hashing and the atomic payload-file write happen before the state lock is taken (file names are content-addressed, so this is safe under concurrency), and a store whose manifest entry is already current is recognized from the manifest alone and skips the clone and commit entirely.
- **Backpressure.** Opportunistic read-through stores are dropped — the in-memory tiers still serve them — when the writer backlog exceeds 1,024 jobs or 128 MiB of queued range payloads, so the queue's latency and memory stay bounded. Coherence-critical work (invalidations, waited inserts) is never dropped.

An enriched directory snapshot embeds each child's metadata. Content, attribute, xattr, link, rename/move, exchange, and remove operations therefore invalidate not only the changed node's own metadata/ranges but every cached directory snapshot containing that node. The persistent implementation commits that node-wide invalidation in one manifest transaction. The adapter applies the same rule to active Finder views and retains metadata for the same 30-second lifetime as the directory snapshot, so a second callback wave cannot fall back to one remote `GetMetadata` request per child.

The cache is not an offline synchronization system. Mount startup still authenticates the server and selects a trustworthy epoch/authorization namespace. Write opens and every namespace/content mutation fail closed while disconnected. No operation journal, queued mutation, merge policy, lock lease, or conflict UI exists.

This is intentionally similar to an ordinary SMB client's read/page cache, not to a synchronization product. SMB is not wholly cache-free: clients use caching, opportunistic locks/leases, and durable handles, while optional products such as Windows Offline Files can add disconnected synchronization. A basic share still needs its server for authoritative namespace and mutation decisions.

Allowing QuickFS to mutate while disconnected would create exactly the merge risks that distinction suggests. Two clients could edit overlapping byte ranges, rename or delete the same object, reuse a pathname, or believe they each hold an exclusive lock. The server might also revoke access, change quota, or begin a new export epoch while a client is away. A safe implementation would need a durable idempotent mutation journal, base revisions and preconditions, stable object identity, reauthentication plus permission/quota/lock revalidation, defined rename/delete conflict rules, and user-visible reconciliation. QuickFS deliberately fails writes closed until such a design exists.

Cold-start offline mounting is a separate problem. Without contacting the server, a client cannot authenticate the current server identity and account, confirm the export epoch, learn that permissions were not revoked, or prove that its cached namespace is current. Serving selected data only after a previously authenticated online mount disconnects keeps stale cache entries from becoming an unauthenticated substitute filesystem.
