# Changelog

## Unreleased

- Make a connection's node working set self-heal instead of failing once it fills. Referencing a node holds one per-connection permit and one global permit; the global pool already evicted an unreferenced cached entry under pressure and retried, but the per-connection pool did not — it returned `TooManyNodes` the instant it filled. Because macOS macFUSE reclaims vnodes lazily (and `auto_xattr` roughly doubles referenced inodes with `._name` sidecars), a single long-lived mount's working set grows monotonically as it browses; with no `Forget` to release permits, a mount that had been up long enough — on the order of an hour of crawling a large library — began failing *every* `getattr`/`lookup` of a new node with `TooLarge: too many known nodes` (surfaced as `EFBIG`) until it was remounted, with no daemon restart able to help. The per-connection pool now evicts one of its own nodes when full — skipping the root and any node backing an open handle — and retries, exactly mirroring the global tier, so the working set is a bounded cache over the larger global registry. Eviction only drops that connection's reference: the node stays in the global registry (four times larger by default) and its stable identifier is unchanged, so the kernel's next access re-tracks it from cache rather than a rediscovery walk and nothing the client cached goes stale. The per-connection memory bound is unchanged (it still holds at most the cap in permits at once), and a listing of a directory larger than the cap now completes by evicting as it scans instead of failing. Server-only; wire protocol and client unchanged (`docs/protocol.md`).

- Cap the in-flight speculative-prefetch convoy at 32 MiB, separate from the read-ahead buffer-depth ceiling. Prefetch permits are held for a fetch's whole lifetime, so the byte pool bounds how much background data is on the wire at once; it was sized to `--read-ahead-max-bytes`, and combined with the 8-concurrent-fetch cap and a large cache block (`--cache-block-kib 8192` → 8 MiB) that put up to 64 MiB of speculation in flight. On the single shared QUIC connection a video demand read then queued behind that convoy whenever another process scanned the volume — Spotlight indexing a freshly mounted share, Quick Look thumbnailing — producing intermittent playback microstutters (reproduced: paced playback under six concurrent bulk readers spiked to ~700 ms). The convoy is now capped independently of buffer depth (`--read-ahead-max-bytes` still sets how deep the window may grow and how much may sit *cached* ahead of the consumer), which dropped the worst-case contended stall to ~130 ms — often zero — with no loss of single-stream throughput. Streams at 1 MiB granularity are unchanged (their 8-fetch cap still binds first); only large-block streams stream a shorter convoy. Client-only; wire protocol and server unchanged.

- Fix first-open video buffering on a low-latency LAN. The adaptive read-ahead window only armed after two in-order reads and started a single block deep, so the opening reads of a freshly opened file had no read-ahead: each cold block was demand-fetched serially at its full per-block latency, and on a fast LAN that serial ramp — not bandwidth — stalled the start of playback (a paced 8 MB/s cold stream stalled 150 ms–1 s at the start, ~1 s when a Finder directory crawl shared the connection). The first read of a handle now primes a shallow speculative read-ahead past the demand block so the second read is a cache hit, and the window arms several blocks deep once a sequential run is confirmed rather than ramping up from one. Priming is gated on there being no tracked stream, so only the cold open primes — a mid-file seek beside a live stream and a crawl's short-lived reads start cold as before, bounding the wasted read-ahead of a one-shot probe. Measured against the same first-open workload the stalls drop to zero (isolated and crawl-contended), with per-read latency under ~75 ms; this is a client-only scheduling change — the wire protocol and server are unchanged. Purely additive to the prefetcher tuned in the video-stutter fix (`docs/caching.md`).

- Add major.minor protocol version negotiation, and make frame compression a negotiated minor feature (this release is **protocol 6.3**, ALPN `quickfs/6`). Every additive protocol change used to force a lockstep flag day, because the ALPN carried the whole version and the per-request check demanded exact equality. The version is now a major.minor pair packed into the existing `u16`: the ALPN is keyed to the major only, so all `6.x` peers connect, and the check accepts any same-major minor and rejects only a different major. `Hello`/`HelloAck` carry each side's exact version — the client sends `Hello` before authenticating and learns the server's, and the server reads the client's off every request — and a minor's optional capabilities are emitted only when the partner advertises support. The upshot: a `6.3` daemon serves a `6.0` client and a `6.3` client talks to a future `6.x` daemon, all without a flag day. Introducing negotiation is itself the one break versus pre-negotiation daemons; after it, additive features (compression here, more later) are graceful minor bumps.

- Compress control frames on the wire (**minor 6.3**, negotiated). Directory views and metadata are highly compressible — repeated names, extensions, and xattr keys — but TLS 1.3 carries no compression, so those bytes went out in full. The four-byte frame length prefix now carries a high-bit flag: when set, the framed body is zstd-compressed (level 3) and the low 31 bits hold its on-wire length; when clear, the body is raw postcard exactly as before. A peer emits a compressed frame only when the partner advertises minor ≥ 6.3 (so an older `6.x` peer that cannot decompress never receives one) *and* the payload is at least 1 KiB *and* it actually shrinks — so pings, single-node metadata, and incompressible bodies are never enlarged, while a realistic media-library directory view compresses roughly 7× (≈86% fewer bytes on the wire, even with high-entropy per-child sizes and timestamps), directly speeding cold crawls over a WAN. Decoding a compressed frame is always supported (the flag is self-describing), so the gate governs the sender, not the receiver. It happens after postcard encoding and after directory-view chunk packing, so the 1 MiB frame limit and chunk budget are still measured against the uncompressed size (compress then fit) and paging is unchanged; on receipt a compressed body is decompressed with its output bounded to `MAX_FRAME_SIZE` as a decompression-bomb guard. Raw file-data bodies (`ReadRange`/`GetXattr`/`WriteRange`) are not control frames and are left untouched.

- Paginate large directories (**minor 6.1**). Previously an enriched directory view had to fit one 1 MiB control frame, so a directory with more than roughly ten thousand entries could not be projected at all — the server returned `TooLarge` and the mount surfaced `EFBIG`, which real media libraries hit. The server now streams a view too big for one frame as a `DirectoryViewStart` header plus frame-sized `DirectoryViewChunk` batches ending in `DirectoryViewEnd`, all over the request's own stream and from one revision-consistent snapshot (no cursor to expire, no cross-page skew), and the client reassembles them transparently — every layer above the transport still receives one `DirectoryView`. Directories that already fit one frame are unchanged. Both peers cap the materialized entry set at `MAX_DIRECTORY_ENTRIES` (1,048,576) as a memory backstop, and the legacy single-frame `ListDirectory` path now returns a clean `TooLarge` error instead of dropping the connection.

- Improve the odds a freshly mounted volume registers usably for Finder/LaunchServices on a host with degraded CoreServices state. Whether `coreservicesd` builds a working file-ID tree for a new volume was measured to be an independent per-mount coin flip that nothing the filesystem reports (statfs, volume capabilities, mount options, cache prewarm) influences and that is not tied to the mountpoint path or volume name; a broken registration is permanent for that mount instance, so the only recovery is a fresh mount. Because the attempts are independent, the mount's existing verify-and-remount retry is the one effective lever, so its budget was raised from 4 to 8 attempts (`VOLUME_REGISTRATION_ATTEMPTS`), lowering the probability of ending on an unusable-for-Finder volume geometrically. The extra retries only run, and only cost time, on a host that is already failing; a healthy host still registers on the first attempt.

- Fix stuttering large-video playback on mounted volumes: track up to four interleaved sequential streams per handle (macOS multiplexes the player, Spotlight, and Quick Look over one FUSE handle; the old single-stream tracker reset its window on every switch), grow the read-ahead window on demand-read stalls instead of a noisy throughput gradient, bound speculation by bytes *and* by concurrent fetches so the window streams instead of convoying, reserve fetch budgets non-blockingly so unfundable blocks are rescheduled rather than becoming never-fetched holes, deadline speculative fetches so a wedged connection cannot pin the slot pool or hang a joining demand read, serve exactly-1 MiB kernel reads from 1 MiB cache blocks, and mount with `-o noreadahead` because macOS kernel read-ahead was measured racing 100+ MB past a paced consumer, thrashing its own speculative pages, and amplifying reads 5.5× (docs/caching.md). Paced 4–8 MB/s playback from cold data: 18–27 s stalled per minute before, zero stalls after; flat-out cold sequential throughput 26.8 → 49.8 MB/s on the same LAN.
