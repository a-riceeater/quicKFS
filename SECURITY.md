# Security policy

Only the latest source revision is supported. Private vulnerability reporting will be enabled before a public release; until then, contact the maintainers privately rather than opening a public issue.

quicKFS is experimental. It assumes a protected persistent server identity, strong user passwords, a private network, and an untrusted client confined to one export root. The selected bootstrap must also be protected: the pairing-code channel, centrally distributed exact pin, deployed enterprise CA, or operating-system trust policy. Mutual-proof pairing, managed pins, standard X.509 chain/name validation, atomic external-identity generations, Argon2id password verification, per-source throttling, global authentication-work bounds, and request/resource limits are implemented. Credential recovery, live-session revocation, distributed-login defense, per-user authorization, audit logging, revocation integration, and independent formal protocol review remain incomplete. Do not expose the prototype to the public Internet without understanding these risks.
